# 短线分时策略优化备忘

本文记录短线分时策略后续优化方向，先作为方案依据，不直接修改策略代码。

## 1. 运行时间治理

### 1.1 现象

当前上午版 workflow 配置为：

```yaml
cron: '20 2 * * 1-5'
```

GitHub Actions 的 cron 使用 UTC 时间，因此该配置对应北京时间周一到周五 `10:20`。但近期实际 scheduled run 经常出现在北京时间 `14:00` 以后，和期望时间窗口明显不一致。

已观察到的实际触发时间包括：

- `2026-07-03 14:04 GMT+8`
- `2026-07-02 14:11 GMT+8`
- `2026-07-01 14:45 GMT+8`
- `2026-06-30 14:33 GMT+8`
- `2026-06-29 15:07 GMT+8`

这类偏差已经超过正常几分钟级延迟，需要在 workflow 和脚本内部都做保护。

### 1.2 可能原因

- GitHub Actions schedule 不保证准点触发，高峰期可能明显延迟。
- workflow 历史上曾经拆分过上午版和下午版；当前已合并为同一个 `intraday` profile，旧 `intraday_pm` 只作为兼容别名。
- GitHub run 使用的是触发时默认分支上的 workflow 文件版本，不能只看当前本地文件判断历史 run 的配置。

### 1.3 建议

1. workflow 开头增加运行审计输出：
   - `github.event_name`
   - `github.event.schedule`
   - `github.sha`
   - UTC 当前时间
   - Asia/Shanghai 当前时间
   - 期望运行窗口

2. 脚本内部增加实际运行窗口保护：
   - 定时运行只允许北京时间 `09:45 ~ 10:45` 继续扫描。
   - 如果 GitHub 延迟到下午才触发，直接生成跳过摘要，不再输出盘中买入信号。
   - 手动触发用于临时复查盘面，应跳过运行窗口判断并强制拉取最新分时数据。

3. 如果继续依赖 GitHub Actions，上午版可以配置多次触发：
   - `09:50`
   - `10:10`
   - `10:30`

   然后在脚本内部按交易日去重，只保留当天第一次有效扫描或最新有效扫描。

4. 对时间敏感的盘中策略，长期更稳的做法是用外部准点调度器触发 `workflow_dispatch`，或者改用云主机 cron 执行。

## 2. 选票优化方向

这里的“选票”指日线候选池预筛，也就是 `build_daily_candidates()` 前后的候选生成与排序逻辑。当前策略已经不是全市场盲扫，而是先用前一交易日日线筛出值得盯盘的股票，再用当天 1 分钟分时确认。

当前选票主要依赖这些维度：

- 均线趋势：`close > ma5 > ma10 > ma20`
- 前日涨幅：按主板 / 20cm 股票区分强度区间
- 换手率：控制低活跃和过热换手
- 成交额：要求放量，但过滤爆量
- 短期动量：`ret5`、`ret10`、`ret20`
- 收盘位置：`close_pos`
- 上影线：`upper_shadow_pct`
- 突破位置：接近或突破前高
- 前一日状态：避免前一日已经过度冲高
- 开盘缺口：过滤前一日高开过猛的结构

### 2.1 先把候选分层，而不是只给一个总分

建议把候选池拆成三类，并在输出中增加 `candidate_type`：

1. 弱转强型
   - 前日涨幅中等偏强。
   - 收盘位置高。
   - 量能放大但不过热。
   - 次日重点看是否突破开盘区间并站稳 VWAP。

2. 龙头加速型
   - 前日接近涨停或强势突破。
   - 对 `ret20`、换手率、成交额倍数要更严格。
   - 次日不能大幅高开直接追，只接受温和高开后的承接确认。

3. 分歧修复型
   - 前日存在一定上影线或盘中分歧，但收盘仍强。
   - 次日必须重新站回 VWAP，并突破开盘区间高点。
   - 适合捕捉换手后再走强的票。

分层的好处是：同样是候选票，不同类型的买点和失效条件不同，后面分时确认可以用不同参数。

### 2.2 增加过热过滤

当前已有 `ret20` 上限，但可以进一步识别加速末端：

- 最近 20 日涨幅过高，同时成交额倍数过高，降权或剔除。
- 最近 5 日涨幅过快，但当日收盘位置没有明显优势，剔除。
- 前日放出天量但收盘没有封在高位，剔除。
- 换手率处于高位，且 `turnover_ratio / turn_ma5` 继续急剧放大，降权。

目标是减少“看起来很强，但实际是兑现日”的候选。

### 2.3 增加连续加速过滤

需要避免接在连续高潮后的尾端：

- 最近 3 日累计涨幅过高时，候选必须有更强的收盘位置和更低的上影线。
- 前一日涨幅接近涨停，且再前一日也大涨时，次日只保留龙头加速型，不放入普通弱转强池。
- 连续两天高开高走后，第三天只观察，不作为默认买入候选。

### 2.4 优化日线分数

当前 `daily_score` 偏重强度，可以改成“强度 + 质量 - 风险”的结构：

强度项：

- 前日涨幅
- 突破幅度
- `ret5`、`ret10`
- 成交额放大

质量项：

- 收盘位置
- 上影线短
- 放量但不过度
- 换手相对均值温和放大

风险扣分：

- `ret20` 过热
- 成交额爆量
- 换手过热
- 前日高开过多
- 前一日已经大涨

这样可以避免总分被单一涨幅或成交额拉高。

### 2.5 增加候选池诊断报表

每次运行建议输出候选池漏斗：

- 股票池总数
- 日线候选数
- 弱转强型数量
- 龙头加速型数量
- 分歧修复型数量
- 因过热剔除数量
- 因放量异常剔除数量
- 因上影线 / 收盘位置不足剔除数量
- 最终参与分钟扫描数量

这比只看最终信号更有用，可以判断问题出在“选票太宽”还是“分时确认太严”。

### 2.6 信号后表现反推选票质量

选票优化最终要靠信号后表现反推。建议记录这些结果：

- 信号后 10 分钟收益
- 信号后 30 分钟收益
- 收盘收益
- 次日开盘收益
- 次日收盘收益
- 是否跌破 VWAP
- 是否跌破开盘区间高点

然后按候选类型聚合：

- 哪类候选胜率最高。
- 哪类候选最容易冲高回落。
- 哪类候选更适合早盘，哪类更适合下午二次确认。

## 3. 建议执行顺序

1. 先加运行时间审计和窗口保护，避免 GitHub 延迟导致策略在错误时段运行。
2. 增加候选池诊断报表，不改变现有选票规则。
3. 在候选输出中增加候选类型和风险标签。
4. 根据 2 到 4 周的信号后表现，再调整日线选票阈值。
5. 最后再做分时参数分层，避免一开始同时改太多变量。

## 4. 散户实战优先级

从 A 股散户交易的角度，最值得优先强化的是“少而准、能执行、回撤小”。

1. 先优化选票，不要先优化买点
   - 只保留高流动性标的。
   - 剔除已经被拉太远、明显过热的票。
   - 保留“强势但未过热”的候选，不把纯情绪高潮票混进来。

2. 把追涨改成确认后再进
   - 开盘区间突破后站稳再看。
   - 回踩 VWAP 不破再看。
   - 如果高开太大，宁可放弃，不硬追。

3. 卖点比买点更重要
   - 固定止损要机械。
   - 移动止盈要早一点介入。
   - 时间止损要明确，避免长时间占用资金。

4. 控制仓位，不要让单票带歪组合
   - 单票仓位不要太重。
   - 同风格票不要同时上太多。
   - 市场弱时自动降仓或空仓。

5. 市场环境要比个股优先
   - 指数弱、情绪弱时减少开仓。
   - 盘中大环境不对时宁可不做。
   - 不要在“个股强、市场烂”的时候硬追。

6. 运行时间必须和交易行为对齐
   - 实际运行时间要审计。
   - 超出窗口自动跳过。
   - 上午、下午策略分开，不要让延迟运行污染信号。

7. 输出要更像交易备忘
   - 票名。
   - 买点。
   - 不追价格。
   - 失效条件。
   - 止损线。
   - 持有时间上限。
